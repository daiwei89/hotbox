Hotbox reduces ML data size so they load faster

ML systems routinely loads GBs to TBs of data for training. Hotbox builds high dimensional schemas of the data of up to billions of dimensions to encode the data in lossless ways. With compression, it can reduce the data size by up to 5x! With Hotbox, ML algorithms use less storage footprint and load data faster. It would work out of box for

  • Classification: logistic regression, SVM, Decision Tree, Neural Network
  • Clustering: K-means, Gaussian Mixture Model
  • Collaborative Filtering, Matrix Factorization, Sparse Coding
  • Many more...(Practically all the ML algorithms are supported!)

Compressed Storage

Most ML practitioners resort to a compact data format called "libsvm" format. This format works particularly well for dataset with high dimensions (many features) but for most data points only very few of the features are relevant (i.e., having non-zero values). Libsvm format exploits this so-called "sparsity" of the dataset and stores only the non-zero values of each data point, and thus is far superior to other formats such as csv for high dimensional sparse dataset.

Here we show that Hotbox has a significant advantage in terms of storage size over even this compact data format. We use KDD Cup 2010 Dataset, which records interactions between students and tutoring systems. The dataset was used to predict student performance on math problems, and has 29 million dimensions and 20 million data point. We found that Hotbox stores it with less than 20% of the storage needed for Libsvm format.

Fast Loading

We applied Hotbox to the largest public dataset in online advertisement. The task is to predict click through rate (CTR) given the search query, user characteristics, ads content, etc, which together form the "feature" of the event. This feature can be very high dimensional. In this case it is 55 million.

We load this dataset from Hotbox to Bosen, a Petuum distributed training system, and found that it's 36% faster than loading the same dataset to Spark from Hive. This can be signficant speedup as data loading time is often significant in training.

Hotbox generates features to improve model performance

In practice, Machine Learning (ML) needs extensive feature engineering to work well. Feature engineering refers to the practice of creating new "views" of the original dataset by combining existing features in complex ways. In fact, the high dimensionality of the KDD dataset above is resulted from heavy feature engineering. As a result, even originally small set of features (e.g., thousands) could become very high dimensional (e.g., billions):

Hotbox supports transforms from one-hot encoding, cartesian products to gradient boosted tree and neural network transforms (by working with external training systems), with many more to come:





Hotbox performs feature transforms over distributed workers (scales linearly with the number of workers) in a streaming fashion (one record at a time). Thus it uses very little memory and is blazing fast!

Why Hotbox?

  • Better Data Storage and Loading: Today's ML datasets could easily exceed the combined memory of trainer machines, especially after feature engineering. In those cases it is unrealistic to cache all datasets in memory as done in Tachyon and Spark. By streaming data from compressed store and applying transforms on the fly, Hotbox pays a small CPU cost but removes any memory limitation caused by large datasets.
  • Ultra High Dimensional: Hotbox can generate up to billions of features efficiently.
  • Petuum Integration: Hotbox works with Petuum systems such as Bosen and Strads. Feature engineering and training in one performant eco-system.
  • Speed: Hotbox transform data on the fly (a.k.a streaming) instead of waiting for all the data to be loaded. No problem if data doesn't fit in memory. Also transformed features are often larger than the raw features. By not storing it we save disk at small compute cost. In contrast, Spark feature transforms only works in batch mode. Dataset exceeding memory size will likely to be very slow.
  • Lightweight: Hotbox encodes data smartly and compress before storing, reducing dataset size by up to 5x. For ML applications whose runtime dominated by disk IO time, this is a huge saving. In our experiment Hotbox-equipped system can load data 36% faster than Spark loading from Hive.
  • Convenience: Simple C++ and Python API to integrate with your ML application. Fully support distributed training and worker failure and elastic scaling.
  • ML support: Come with a library of common transforms for ML applications that configurable for your app.

Performance

Large-scale Training

Hotbox is easy to use with existing training systems. We integrate Hotbox with Petuum Bosen, a distributed training system, and Spark. We come back to the KDD Cup example and train a Logistic Regression model, which is the winning solution of the challenge. We record the end-to-end time (including data loading and training) for Petuum Bosen and Spark, both equipped with Hotbox, to reach a sufficiently good model (measured by Logistic Regression objective).

We observe that Petuum Bosen outperforms Spark by a factor of 10. This is mainly because Petuum Bosen uses a synchronization model more suited for machine learning tasks (see here for more details). Hotbox here nonetheless reduces data management complexity as well as loading time for the ML systems.
